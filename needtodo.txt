Packaging-Critical Risks

External binaries must exist on PATH:
ffmpeg/ffprobe in letra_toolkit.py:133 and letra_toolkit.py:157.
MFA via conda run -n mfa in _mfa_aligner.py:526 and _mfa_aligner.py:188.
macOS-specific UI behavior:
osascript calls in gui.py:40 and frase.py:181.
Argos timeout uses SIGALRM in argos_translate.py:29, argos_translate.py:31, argos_translate.py:35, which is not Windows-safe.
Output location is relative to working directory:
GUI sets worker cwd to project root in gui.py:459.
Worker creates output folder by basename in _worker.py:99 and _worker.py:100.
This is likely to fail once installed under read-only app locations.
Installer tooling is currently mac-only and incomplete in setup_mac.py:9, setup_mac.py:17, setup_mac.py:22.
Dependencies are heavy and compiled (requirements.in:1, requirements.in:2, requirements.in:3, requirements.in:15, requirements.in:18), so pure pip one-click installers will be fragile across OSes.
Best Packaging Strategy

Use Conda-first distribution with platform-native installers.
Build one environment spec, generate per-OS lockfiles, and package each OS artifact in CI.
Recommended stack:
conda-lock for reproducible envs on macOS/Windows/Linux.
micromamba bootstrap in installer runtime.
constructor (or equivalent native wrapper) to produce pkg/exe/sh installers.
Keep one source, produce three installers. That is the practical meaning of system-agnostic for this app class.
Why this is best for your project:

You already require MFA through Conda.
ffmpeg and many compiled libs are easier and more reliable through Conda-forge than mixed pip wheels.
You avoid brittle path assumptions from user Python installs.
You can still add a lightweight app launcher and desktop shortcuts.
